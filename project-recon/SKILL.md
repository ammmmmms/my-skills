# Project Recon

用于大型或陌生项目的代码侦察。目标是先定位，再解释，再给出下一步建议；不是直接实现需求，也不是泛泛阅读整个仓库。

## 核心原则

- **证据优先**：每个结论都必须能落到文件、符号、字符串、配置、测试或调用关系上。
- **先广后深**：先用低成本搜索收敛候选，再沿最强证据深挖；不要一开始就读大文件。
- **不预设结构**：不要假设功能一定在页面、路由、store、service 中，也要检查 utils、配置、脚本、常量、测试、生成代码入口等位置。
- **限制分叉**：同时最多保留 2 到 3 条候选路线，其余只简短记录，不继续展开。
- **及时止损**：证据不足时停止搜索并向用户提问，不允许无边界盲搜。

## 适用任务

- 找某个功能或交互相关文件
- 梳理调用链、数据流、模块职责
- 判断改某个需求大概率会动到哪里

## 不适用任务

- 直接实现功能或修 bug
- 输出框架最佳实践或重构规范
- 无目标地通读整个项目

## 工作流

按顺序执行，优先收敛。

### 1) 重述任务并提取搜索锚点

先把用户目标压缩成一句技术任务，然后提取可搜索锚点。

优先使用这些锚点：

- 页面文案、按钮文案、报错文案、空态文案
- 函数名、组件名、类名、模块名、配置 key
- 接口名、字段名、事件名、埋点名、日志片段
- 测试名、快照名、fixture 名

如果用户已经提出新需求，额外提取：

- 想改的行为
- 期望影响的页面或模块
- 明确不允许被影响的范围

### 2) 广搜候选，不读大块无关内容

先搜索锚点，建立候选集合，再决定深挖路径。

- 优先找字符串命中、符号定义、导入关系、导出关系、配置引用
- 先看文件名、目录名、调用点密度，再决定是否打开文件
- 先缩小到少量候选，再读代码细节

如果多个锚点互相矛盾，以更具体、更可验证的锚点为准。

### 3) 深挖最强证据路径

只沿最可能的主路径继续追。

优先确认：

- 功能入口或触发点
- 中间调度层或桥接层
- 真正生效的实现点
- 相关状态来源、配置来源、数据转换点
- 测试或文档中是否有旁证

不要为了“完整”而展开所有分支。主路径成立即可。

### 4) 总结模块职责和调用关系

在收敛后，用简洁语言解释：

- 这个模块负责什么，不负责什么
- 输入来自哪里
- 输出影响哪里
- 哪些文件是主实现，哪些只是包装、透传、适配或测试

### 5) 如果用户已提出改动需求，再补变更建议

只在已经找到主路径后给建议，不要在定位不清时直接推修改方案。

需要评估：

- 最可能的修改点
- 可能受影响的上下游
- 容易漏改的配置、类型、测试、埋点、文案
- 最小验证范围

## 必须停止并询问用户的情况

出现以下任一情况时，停止继续搜索，改为提问：

- 用户只给了业务概念，没有任何技术锚点
- 搜索结果出现多个强候选，短时间内无法排除
- 连续两轮深挖都没有新增强证据
- 功能疑似由配置、插件、代码生成或远端下发驱动，常规代码路径不成立
- 用户目标不明确，无法判断是要“找到实现位置”还是“分析改动方案”

提问要短且收敛，优先问：

- 这个功能对应哪个页面、按钮、弹窗或文案？
- 有没有接口名、字段名、报错信息、日志片段或配置 key？
- 你现在更需要“找到实现位置”还是“分析修改方案”？
- 这几个候选路径里，你更关注哪一个？

## 输出契约

每次交付都按这个结构输出，压缩搜索过程，只保留结论和证据：

### 任务重述

一句话说明这次侦察的目标。

### 搜索锚点

列出本次实际使用的锚点；没有锚点时明确写出缺失。

### 关键证据

列出支持结论的直接证据，例如：

- 某字符串或文案命中
- 某符号定义与调用关系
- 某配置项与运行逻辑的关联
- 某测试覆盖了该行为

### 最相关文件

只列 Top 3 到 Top 5，并说明每个文件为何相关。

### 调用链 / 数据流

只写主路径，不展开次要分支。

### 模块职责总结

说明主模块、辅助模块、包装层分别做什么。

### 不确定点

明确写出尚未确认的部分，以及为什么没有确认。

### 建议下一步

如果定位已经足够清楚，给主 agent 一个最合理的下一步动作。

如果用户已经提出新需求，追加以下两段：

### 推荐修改点

给出最可能需要改的文件或层级，并说明原因。

### 潜在影响面与验证建议

只列最值得检查的影响面和最小验证范围。
